Garanti BBVA Virtual POS Laravel Paketi

Bu paket Laravel projelerinde Garanti BBVA GVP sanal POS XML, 3D, OOS, GarantiPay, puan, güvenlik ve sorgu akışlarını üretmek için kullanılır.

Paket davranışı repo içindeki resmi Help/GVP örnekleriyle fixture/test seviyesinde karşılaştırılmıştır. Ayrıntılı madde madde kanıt listesi için GVP Compatibility Checklist dosyasını kullanın.

Kurulum

composer require developertugrul/garanti-pos
php artisan vendor:publish --provider="Developertugrul\GarantiPos\GarantiPosServiceProvider"

GARANTI_POS_REFUND_PASSWORD veya GARANTI_POS_PROV_OOS_PASSWORD boşsa paket GARANTI_POS_PROV_PASSWORD değerini kullanır. Formda gönderilen terminalprovuserid hangi kullanıcıysa hash aynı kullanıcının şifresiyle hesaplanır.

Hash ve Güvenlik

  • Legacy Help/GVP varsayılanı: SecurityData = SHA1(provPassword + 9 haneli terminalId).
  • Legacy XML hash: orderId + terminalId + cardNumber + amount + securityData.
  • Legacy 3D/OOS form hash: terminalId + orderId + amount + successUrl + errorUrl + txntype + installment + storeKey + securityData.
  • GarantiPay form hash'i kesin olarak txntype=gpdatarequest değeriyle üretilir; satış tipi txnsubtype=sales olarak gider.
  • Güncel portal SHA512 akışı için GARANTI_POS_HASH_ALGORITHM=sha512 ve genellikle GARANTI_POS_API_VERSION=512 kullanılabilir.
Repo içindeki resmi Help/GVP örnekleri SHA1 akışını kullanır. Güncel developer portal ortak ödeme formunda apiversion=512 ve farklı hash modeli vardır; terminaliniz hangi akışa tanımlandıysa o konfigürasyon seçilmelidir.

Sorgular

  • orderInquiry(): Type=orderinq
  • orderHistoryInquiry(): Type=orderhistoryinq
  • orderListInquiry(): StartDate/EndDate Order altında, ListPageNum Transaction altında
  • batchInquiry(): BatchNum/ListPageNum Transaction altında
  • binInquiry(): Type=bininq
  • campaignCodeInquiry(): resmi yazımıyla CampaingCode
  • settlementInquiry(): root SettlementInq bloğu

3D Dönüş Kontrolü

validate3DHash() sadece callback hash kontrolünü yapar. Ödeme kabul kararı için parse3DResponse() ile hash, mdstatus, procreturncode, sipariş ve tutar kontrolleri ayrı ayrı okunmalıdır.

$check = GarantiPos::parse3DResponse($request->all(), [
    'order_id' => $order->number,
    'amount' => $order->amount_minor,
]);

if (!$check['hash_valid'] || !$check['md_status_accepted'] || !$check['proc_return_success']) {
    abort(400);
}

$response = GarantiPos::pay3DModel(
    ['order_id' => $order->number, 'amount' => $order->amount_minor],
    [],
    $request->all()
);

Test ve Doğrulama

composer install
vendor/bin/phpunit --configuration phpunit.xml.dist

WSL/XAMPP ortamında:

cmd.exe /c "cd /d C:\xampp\htdocs\libs\garanti-pos && C:\xampp\php\php.exe vendor\bin\phpunit --configuration phpunit.xml.dist"

Varsayılan testler bankaya canlı istek atmaz; hash, XML, form alanları ve HTTP payload üretimi fixture/stub ile doğrulanır. Canlı banka testi gerçek merchant bilgileriyle ayrıca manuel smoke olarak yapılmalıdır.
